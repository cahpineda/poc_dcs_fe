<objective>
Add a small colored dot at the top-left of each occupied seat cell indicating passenger gender (M=blue, F=pink, U=orange for unaccompanied minor / monk). Match cloud_2's `.gender_label_background_*` overlay using POC design-token-aligned colors. Strict TDD — start at the SeatCell render boundary since the visual badge is the user-observable contract.
</objective>

<context>
@src/presentation/components/SeatCell.tsx
@src/domain/seat/Seat.ts
@src/infrastructure/adapters/mappers/seatPlanMapper.ts
@src/styles/components.css
@src/infrastructure/adapters/MockSeatPlanAdapter.ts

Plans 09-01 and 09-02 established the overlay pattern (Seat field -> mapper -> SeatCell -> CSS). This plan applies it once more for the gender badge — the third and final overlay in Phase 9.

Cloud_2 uses three classes:
- `.gender_label_background_male`   (blue)
- `.gender_label_background_female` (pink)
- `.gender_label_background_monk`   (orange) — used for both monks AND unaccompanied minors in cloud_2

POC adopts the values:
- gender 'M' -> `.seat_gender_male`   #1565c0
- gender 'F' -> `.seat_gender_female` #e91e63
- gender 'U' -> `.seat_gender_unaccompanied` #f57c00

The badge is purely decorative — agents read passenger lists in the side panel for accessibility-required gender info. The badge is `aria-hidden` to avoid duplicate screen-reader noise.

Position: top-left corner of the seat cell. The block indicator (Plan 09-02) sits at bottom-left; the infant indicator at bottom-right. Top-right is reserved for the existing price badge. Top-left is therefore the only free corner.
</context>

1. Define gender type alias near the top of SeatCell.tsx (or import if a shared type makes sense — keep it local for now to limit blast radius):
   ```ts
   type Gender = 'M' | 'F' | 'U';

   const GENDER_CLASS: Record<Gender, string> = {
     M: 'seat_gender_male',
     F: 'seat_gender_female',
     U: 'seat_gender_unaccompanied',
   };
   ```

2. Extend `SeatCellProps`:
   ```ts
   export interface SeatCellProps {
     seatNumber: string;
     status: SeatStatus;
     isExitRow?: boolean;
     isSelected?: boolean;
     price?: number;
     passengerInitials?: string | null;
     hasInfant?: boolean;
     blockNote?: string;
     gender?: Gender | null;
     onSelect: (seatNumber: string) => void;
   }
   ```

3. Update render — insert the badge as the FIRST child of the button so it stacks below textual content if any z-index conflict arises (but absolute positioning makes order largely irrelevant):
   ```tsx
   {gender && (
     <span
       className={`seat_gender_badge ${GENDER_CLASS[gender]}`}
       aria-hidden="true"
     />
   )}
   {showInitials ? passengerInitials : seatNumber}
   {/* ... existing price / infant / block overlays ... */}
   ```

WHY: Using a lookup table for the modifier class scales cleanly if a 4th gender code appears later. `aria-hidden="true"` keeps assistive tech from announcing what amounts to a visual color marker.

AVOID: Do NOT use inline styles for the colors — keep the modifier class pattern so designers can re-skin via CSS without touching components. Do NOT include the badge in click-target area math; it's decorative and `pointer-events: none` (added in Task 3.3) keeps it out.

## Task 3.5 — Implement Seat.gender, mapper, SeatRow wiring, mock data

**files:**
- src/domain/seat/Seat.ts
- src/infrastructure/adapters/mappers/seatPlanMapper.ts
- src/presentation/components/SeatRow.tsx
- src/infrastructure/adapters/MockSeatPlanAdapter.ts

**action:**

1. **Seat.ts** — extend SeatProps and add a runtime guard for the gender code:
   ```ts
   export type SeatGender = 'M' | 'F' | 'U';

   interface SeatProps {
     seatNumber: SeatNumber;
     status: SeatStatus;
     cabinClass: string;
     passengerName?: string;
     hasInfant?: boolean;
     blockNote?: string;
     gender?: SeatGender | null;
   }
   ```

   In `Seat.create`, normalize:
   ```ts
   const validGenders: ReadonlySet<SeatGender> = new Set(['M', 'F', 'U']);
   const gender: SeatGender | null =
     props.gender && validGenders.has(props.gender as SeatGender)
       ? (props.gender as SeatGender)
       : null;
   ```

   Add field + getter:
   ```ts
   private readonly _gender: SeatGender | null;
   // ...constructor stores it...
   get gender(): SeatGender | null { return this._gender; }
   ```

   Update `withStatus` to preserve `_gender`.

2. **seatPlanMapper.ts** — pass DTO gender through:
   ```ts
   Seat.create({
     // ... existing fields ...
     gender: typeof s.gender === 'string' ? (s.gender as 'M' | 'F' | 'U') : undefined,
   });
   ```
   The runtime guard in `Seat.create` will discard invalid codes, so the mapper doesn't need to validate.

3. **SeatRow.tsx** — pass to SeatCell:
   ```tsx
   <SeatCell
     // ... existing props ...
     gender={seat.gender}
   />
   ```

4. **MockSeatPlanAdapter.ts** — assign gender values to occupied mock seats:
   ```ts
   Seat.create({ seatNumber: SeatNumber.create('1B'), status: 'occupied', cabinClass: 'Y', passengerName: 'DEMO PAX', gender: 'M' }),
   // ...
   Seat.create({ seatNumber: SeatNumber.create('2C'), status: 'exit_row_occupied', cabinClass: 'Y', passengerName: 'JANE ROE', gender: 'F' }),
   Seat.create({ seatNumber: SeatNumber.create('2F'), status: 'infant_occupied', cabinClass: 'Y', passengerName: 'BABY OK', hasInfant: true, gender: 'U' }),
   ```

WHY: A `Set` lookup for valid genders is constant-time and easy to extend if the spec adds codes (e.g., 'X' for unspecified). Returning `null` for invalid input keeps the failure mode predictable in the UI (no badge rendered).

AVOID: Do NOT crash on unknown gender codes. Production data sometimes contains legacy codes; a missing badge is preferable to a render error. Do NOT widen the `Gender` type in SeatCell to accept arbitrary strings — keep the union narrow and let the domain layer normalize.
